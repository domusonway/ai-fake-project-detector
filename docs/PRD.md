# ai-fake-project-detector · PRD

---

## 产品定位
这是一个 **GitHub-first 的 AI 项目验证与风险提示产品**。
V1 已交付以 Flask 为主入口的 Web + JSON API 体验，用结构分析、同类项目对比、可解释风险评分、历史快照和补充反馈，帮助用户判断一个热门 AI 开源仓库是否存在“传播大于实质”的风险。

---

## V1 已交付目标
- 输入 GitHub 仓库 URL，触发一次持久化分析并生成解释性报告
- 通过 **同类项目对比** 提升评分可信度，而不是只看单仓库绝对值
- 提供 **已分析仓库范围内** 的排名、搜索、筛选与排序能力
- 保存 **分析快照 / 分数历史**，支持复查同一仓库的变化
- 提供 **登录后可提交的轻量反馈**，作为补充参考，不直接主导评分或排名

---

## 目标用户
- 想快速判断热门 AI 开源项目是否有真实交付基础的开发者
- 需要高效筛选值得跟进仓库的研究者、投资分析者、产品经理
- 关注“项目是否有实质内容与可验证证据”的社区用户

---

## 核心用户问题
用户需要一个统一工具去回答以下问题：
- 这个仓库是否具备基本的交付实质？
- 它与同类项目相比，风险在哪里？
- 这个结论基于哪些可见证据？
- 它的风险判断是否随着重复分析而发生变化？
- 社区补充反馈与系统核心评分分别代表什么？

## V1 范围

### 已交付能力
- GitHub 作为 **唯一 V1 数据源**
- Flask 作为 **canonical V1 surface**，同时提供 HTML 页面与 JSON API
- 单仓库分析入口与重复分析入口
- 仓库结构、README、代码/文档比例、基础活跃度等可解释特征提取
- **规则优先（rule-first）** 的同类项目召回与 baseline 摘要
- 可解释风险评分：总分、风险等级、分维度风险、证据卡片、保护性说明
- 持久化分析快照、版本化历史记录、详情页与详情 API
- **已分析仓库** 排名页与排名 API
- 排名页 / API 的搜索、筛选、排序与透明披露
- 登录态轻量反馈：去重、限流、补充展示，与核心评分分离
- 端到端集成导航：分析页 → 详情/历史 → 排名 → 反馈

### 明确不纳入 V1（延期）
- X / Twitter 数据接入
- 匿名投票或匿名反馈
- 公共开放 API
- 作者申诉 / appeals 流程
- 评论区
- 高级反欺诈 / 高复杂度作弊检测

## 产品原则
- **解释优先**：不能只给总分，必须展示原因与证据
- **对比优先**：评分必须结合 peer baseline，而不是单仓库孤立判断
- **验证语言优先**：面向用户使用“风险 / 可信度 / 验证 / 实质度”语言，不做定罪式表述
- **反馈补充原则**：用户反馈可见但不直接改写核心风险分，也不成为排名权威
- **范围收敛原则**：V1 不依赖 X / Twitter，不依赖匿名社区治理，不依赖开放 API
- **已分析范围原则**：排名、筛选、排序只覆盖系统已分析并持久化的仓库

---

## 评分与结论表达
- V1 输出 `fake_risk_score`（0–100），**分数越高表示风险越高**
- 所有 `dimension_scores` 保持同一语义：越高表示该维度风险越高
- 评分结果必须包含：
  - 总体风险分
  - 风险等级 / risk band
  - 分维度解释
  - 证据卡片
  - peer baseline 摘要
  - guardrail notes / 保护性说明
  - 历史记录入口

### 风险分级
- 0–20：低风险，可信度较高
- 21–40：轻度风险，建议持续观察
- 41–60：中度风险，需要核查关键证据
- 61–80：高风险，存在明显“传播大于实质”信号
- 81–100：极高风险，需谨慎看待

### 评分 guardrails
- 早期但真实的项目不能仅因“不成熟”被高风险化
- 研究型原型若证据充分，应降低误伤
- 所有高风险判断必须附带可见证据
- peer uncertainty 与小样本 baseline 需要显式披露，而不是静默放大惩罚

---

## 排名、历史与反馈边界

### 排名边界
- 排名只覆盖 **系统已经分析并持久化的仓库**
- 不是“全 GitHub 榜单”
- 排名依据是最新持久化风险快照与可验证元数据
- 反馈摘要可展示，但 **feedback is supplemental and never authoritative for rank ordering**

### 历史边界
- 同一仓库重复分析后会生成新的 snapshot version
- 详情页与 API 展示最新快照 + 按时间倒序的历史记录
- 历史用于复查变化，不回写旧快照

### 反馈边界
- 反馈必须是 **authenticated / 登录态**
- 反馈是 **lightweight / 轻量** 的 `support` / `disagree` / `needs_review` 信号
- 反馈是 **supplemental / 补充层**，不直接成为风险分权威来源
- V1 已实现基础去重、时间窗口去重、简单限流与掩码展示

## 关键页面与接口
- 分析提交页：输入 GitHub URL 并触发分析
- 结果页：查看报告摘要并跳转到详情/历史或排名
- 仓库详情页 / API：查看最新报告、历史、peer 披露、补充反馈摘要
- 排名页 / API：浏览已分析仓库、搜索、筛选、排序、透明披露
- 反馈入口：登录用户提交轻量反馈并查看汇总

---

## 成功标准
- 用户可完成“输入 GitHub 仓库 → 查看解释性报告 → 查看同类对比 → 查看历史 → 浏览排名 → 提交补充反馈”的闭环
- 用户能理解“为什么存在风险提示”，而不是只看到结论
- 排名页明确只覆盖已分析仓库，并支持搜索 / 筛选 / 排序
- 历史记录可以反映重复分析后的变化
- 登录反馈被清晰展示为补充信息，而非核心评分权威

---

## 验收标准
- 输入有效 GitHub URL 后可生成并持久化结构化报告
- 报告包含总分、风险等级、分维度分、证据卡片、peer baseline 摘要、guardrail notes
- 详情页 / API 可查看最新快照与历史版本
- 排名页 / API 明确只展示已分析仓库，并支持搜索 / 筛选 / 排序
- 反馈流程要求登录，具备基础去重与限流，且不直接改写核心分数或排名依据
- 集成 / 回归测试覆盖分析、详情、排名、反馈与跨页面导航
